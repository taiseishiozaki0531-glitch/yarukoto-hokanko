# 要件定義書：やること保管庫

## 1. プロジェクト概要

### 1.1 アプリ名

やること保管庫

### 1.2 アプリの目的

「やること保管庫」は、読書・動画・教材・人間関係・買い物など、日常生活の中で「あとでやろう」と思ったまま放置されがちなものを一箇所に集め、次に取るべき行動まで整理するWebアプリである。

単なるToDoリストではなく、明確な締切がないために放置されやすい「後回し案件」を管理することを目的とする。

### 1.3 解決したい課題

ユーザーは日常生活の中で、以下のようなものを多く抱えている。

- 読みたい本
- 見たい動画
- 途中で止まっている教材
- 返信したいLINE
- 会う予定を立てたい友人
- 買いたいもの
- 後で確認したいURL
- いつかやろうと思っている小さなタスク

しかし、これらは以下のように情報が分散しやすい。

- 本はAmazonのほしい物リスト
- 動画はYouTubeの「後で見る」
- 教材はブックマーク
- 人間関係の用事はLINEやInstagram
- 思いつきはメモアプリ
- 買い物候補はスクリーンショットや検索履歴

その結果、ユーザーは「あとでやろう」と思ったことを忘れたり、途中で止まっているものを把握できなくなったりする。

本アプリは、これらの後回し案件を一箇所に保存し、カテゴリ・状態・優先度・次にやることを整理できるようにする。

---

## 2. 対象ユーザー

### 2.1 主な対象ユーザー

大学生・高校生・若手社会人など、日常的に学習・趣味・人間関係・買い物などの小さなタスクを多く抱えている個人ユーザー。

### 2.2 想定ペルソナ

#### ペルソナ：田中悠斗

- 年齢：19歳
- 性別：男性
- 所属：私立大学1年生
- 使用端末：
  - 普段はスマホ
  - 作業時はMacBook
- 生活状況：
  - 授業、サークル、友人関係、アルバイト、個人学習で予定が多い
  - 興味のある本・動画・教材を見つけるが、後で見ようと思って忘れやすい
  - NotionやGoogleカレンダーは使ったことがあるが、細かく管理するのは面倒に感じる
- よく後回しにするもの：
  - 読みたい本
  - 見たいYouTube動画
  - 途中で止まっているオンライン教材
  - 返信しなければいけないLINE
  - 友人への誕生日プレゼント
  - 買いたい服やガジェット
- 悩み：
  - 「あとでやる」と思ったものが散らばって忘れる
  - 何が途中で止まっているのか分からない
  - 緊急ではないが、放置すると損するものを管理できていない
  - ToDoリストに入れるほど明確な締切がないものを整理しづらい
- 求める価値：
  - 後回しにしているものを一箇所で見たい
  - カテゴリごとに整理したい
  - 「次に何をすればいいか」だけすぐ分かるようにしたい
  - 入力項目は多すぎず、シンプルに登録したい

## 3. 利用シーン

### 3.1 読書管理

ユーザーは読みたい本や途中まで読んだ本を登録する。

例：

- タイトル：民主主義の死に方
- カテゴリ：読書
- 総量：320
- 現在の進捗：80
- 単位：ページ
- 次にやること：第3章を20ページ読む
- ステータス：途中
- 優先度：高

### 3.2 動画管理

ユーザーは後で見たい動画や途中まで見た動画を登録する。

例：

- タイトル：Supabase入門動画
- カテゴリ：動画
- URL：https://example.com
- 総量：45
- 現在の進捗：20
- 単位：分
- 次にやること：認証機能の章を15分見る
- ステータス：途中

### 3.3 教材管理

ユーザーは進めたい教材やオンライン講座を登録する。

例：

- タイトル：Vibe Coding教材
- カテゴリ：教材
- 総量：10
- 現在の進捗：3
- 単位：章
- 次にやること：Supabase連携の章を進める
- ステータス：途中
- 優先度：高

### 3.4 人間関係タスク管理

ユーザーは返信・連絡・約束・プレゼントなど、人間関係に関する後回し案件を登録する。

例：

- タイトル：AさんにLINEを返す
- カテゴリ：人間関係
- 相手の名前：Aさん
- 連絡手段：LINE
- 次にやること：週末の予定について返信する
- 期限：2026-05-20
- ステータス：未着手
- 優先度：高

### 3.5 買い物候補管理

ユーザーは買いたいものや検討中の商品を登録する。

例：

- タイトル：黒のトートバッグ
- カテゴリ：買い物
- URL：商品ページURL
- メモ：大学用。予算1万円以内。
- ステータス：保留
- 優先度：中

## 5. 前提条件

### 5.1 技術前提

- フロントエンドはNext.jsを使用する
- 言語はTypeScriptを使用する
- スタイリングはTailwind CSSを使用する
- データベースと認証にはSupabaseを使用する
- デプロイにはVercelを使用する
- まずは個人利用を想定したWebアプリとして作成する

### 5.2 UI前提

- PCとスマホブラウザの両方で使えるレスポンシブデザインにする
- 最初は複雑なデザインよりも、見やすさと操作の分かりやすさを優先する
- 1画面あたりの操作数を少なくし、初心者でも迷わず使えるUIにする

### 5.3 データ設計の前提

初期リリースでは、読書・動画・教材・人間関係・買い物をすべて「アイテム」として扱う。

カテゴリごとに別テーブルを作るのではなく、まずは共通のitemsテーブルで管理する。

想定する主なデータ項目は以下である。

- id
- user_id
- title
- category
- status
- priority
- next_action
- due_date
- url
- memo
- total_amount
- current_amount
- amount_unit
- person_name
- contact_method
- is_today
- created_at
- updated_at

## 6. 提供価値

本アプリが完成すると、ユーザーは以下の価値を得られる。

1. 後回しにしているものを一箇所で管理できる
2. 読書・動画・教材・人間関係・買い物をカテゴリ別に整理できる
3. 「次にやること」が明確になり、再開しやすくなる
4. 途中で止まっているものの進捗を把握できる
5. 今日やるべきものを選び、行動に移しやすくなる
6. 期限切れのものを見落としにくくなる
7. 情報が散らばるストレスを減らせる

# 7. 機能要件

## 7.1 優先度P0：MVPで必須の機能

### 1. ユーザー認証機能

#### 説明

ユーザーは自分専用のデータを管理するために、アカウント登録・ログイン・ログアウトができる。

#### 目的

他のユーザーのデータと混ざらないようにし、個人の後回し案件を安全に保存するため。

#### EARS形式

When ユーザーがアプリにアクセスしたとき、the system shall 未ログインの場合はログイン画面を表示する。

When ユーザーがメールアドレスとパスワードを入力したとき、the system shall Supabase Authを用いてログイン処理を行う。

#### 受け入れ条件

- [ ] ユーザーはメールアドレスとパスワードで新規登録できる
- [ ] ユーザーはメールアドレスとパスワードでログインできる
- [ ] ユーザーはログアウトできる
- [ ] 未ログインユーザーはアイテム一覧や登録画面にアクセスできない
- [ ] ログイン中のユーザーは自分のアイテムのみ閲覧できる

---

### 2. アイテム登録機能

#### 説明

ユーザーは、後回しにしている読書・動画・教材・人間関係・買い物などをアイテムとして登録できる。

#### 目的

頭の中や複数アプリに散らばっている「あとでやること」を一箇所に保存するため。

#### 入力項目

必須項目：

- タイトル
- カテゴリ
- ステータス
- 優先度
- 次にやること

任意項目：

- 期限
- URL
- メモ
- 総量
- 現在の進捗
- 単位
- 相手の名前
- 連絡手段

#### カテゴリ候補

- 読書
- 動画
- 教材
- 人間関係
- 買い物
- その他

#### ステータス候補

- 未着手
- 途中
- 保留
- 完了
- やめた

#### 優先度候補

- 高
- 中
- 低

#### EARS形式

When ユーザーがアイテム追加画面を開いたとき、the system shall アイテム登録フォームを表示する。

When ユーザーが必須項目を入力して保存ボタンを押したとき、the system shall 入力内容をSupabaseに保存する。

#### 受け入れ条件

- [ ] タイトルは必須入力である
- [ ] カテゴリは選択式である
- [ ] ステータスは選択式である
- [ ] 優先度は選択式である
- [ ] 次にやることは必須入力である
- [ ] URLは任意入力である
- [ ] 期限は任意入力である
- [ ] メモは任意入力である
- [ ] 総量・現在の進捗・単位は任意入力である
- [ ] 保存後、アイテム一覧画面またはダッシュボードに反映される
- [ ] 入力エラーがある場合、ユーザーに分かる形でエラーメッセージを表示する

---

### 3. カテゴリ別の入力補助機能

#### 説明

ユーザーがカテゴリを選択したとき、そのカテゴリに合った補助項目を表示する。

#### 目的

不要な入力欄を減らし、カテゴリごとに使いやすい登録フォームにするため。

#### カテゴリ別の想定項目

読書：

- 総ページ数
- 現在のページ
- 単位：ページ

動画：

- URL
- 動画時間
- 視聴済み時間
- 単位：分

教材：

- 教材URL
- 全体の章数
- 現在の章
- 単位：章

人間関係：

- 相手の名前
- 連絡手段
- 期限
- メモ

買い物：

- URL
- メモ
- 優先度
- 期限

#### EARS形式

When ユーザーがカテゴリを選択したとき、the system shall 選択されたカテゴリに応じて関連する入力欄を表示する。

#### 受け入れ条件

- [ ] 読書カテゴリではページ数に関する入力欄を表示する
- [ ] 動画カテゴリではURLと動画時間に関する入力欄を表示する
- [ ] 教材カテゴリでは章数や教材URLに関する入力欄を表示する
- [ ] 人間関係カテゴリでは相手の名前と連絡手段に関する入力欄を表示する
- [ ] 買い物カテゴリではURLとメモ欄を表示する
- [ ] カテゴリを変更しても、共通項目の入力内容は保持される

### 4. アイテム一覧表示機能

#### 説明

ユーザーは登録したアイテムをカード形式またはリスト形式で一覧表示できる。

#### 目的

現在抱えている後回し案件を一目で把握できるようにするため。

#### 表示項目

一覧カードには以下を表示する。

- タイトル
- カテゴリ
- ステータス
- 優先度
- 次にやること
- 期限
- 進捗率
- URLボタン
- 詳細・編集ボタン

#### EARS形式

When ログイン中のユーザーが一覧画面を開いたとき、the system shall そのユーザーが登録したアイテムを一覧表示する。

#### 受け入れ条件

- [ ] 登録済みアイテムがカード形式またはリスト形式で表示される
- [ ] 各アイテムにタイトルが表示される
- [ ] 各アイテムにカテゴリ・ステータス・優先度が表示される
- [ ] 各アイテムに次にやることが表示される
- [ ] 期限が設定されている場合は期限が表示される
- [ ] 進捗情報がある場合は進捗率が表示される
- [ ] URLが登録されている場合はURLを開くボタンが表示される
- [ ] アイテムが0件の場合は空状態メッセージを表示する

### 5. アイテム詳細表示機能

#### 説明

ユーザーは一覧から個別アイテムを選択し、詳細情報を確認できる。

#### 目的

一覧では表示しきれないメモやURL、進捗、連絡手段などを確認するため。

#### EARS形式

When ユーザーが一覧画面でアイテムを選択したとき、the system shall 選択されたアイテムの詳細情報を表示する。

#### 受け入れ条件

- [ ] タイトル、カテゴリ、ステータス、優先度が表示される
- [ ] 次にやることが表示される
- [ ] メモが登録されている場合は表示される
- [ ] URLが登録されている場合はリンクとして表示される
- [ ] 進捗情報が登録されている場合は表示される
- [ ] 人間関係カテゴリの場合、相手の名前と連絡手段が表示される
- [ ] 詳細画面から編集画面に移動できる
- [ ] 詳細画面から一覧画面に戻れる

### 6. アイテム編集機能

#### 説明

ユーザーは登録済みアイテムの内容を編集できる。

#### 目的

進捗状況や次にやることが変わった場合に、情報を更新できるようにするため。

#### EARS形式

When ユーザーが編集ボタンを押したとき、the system shall 現在の登録内容が入力済みの編集フォームを表示する。

When ユーザーが編集内容を保存したとき、the system shall 該当アイテムのデータを更新する。

#### 受け入れ条件

- [ ] 既存の登録内容がフォームに反映される
- [ ] ユーザーはタイトルを編集できる
- [ ] ユーザーはカテゴリを編集できる
- [ ] ユーザーはステータスを編集できる
- [ ] ユーザーは優先度を編集できる
- [ ] ユーザーは次にやることを編集できる
- [ ] ユーザーは期限、URL、メモ、進捗情報を編集できる
- [ ] 保存後、一覧画面と詳細画面に変更内容が反映される

### 7. アイテム削除機能

#### 説明

ユーザーは不要になったアイテムを削除できる。

#### 目的

誤って登録したものや不要になった後回し案件を整理するため。

#### EARS形式

When ユーザーが削除ボタンを押したとき、the system shall 削除確認ダイアログを表示する。

When ユーザーが削除を確定したとき、the system shall 該当アイテムを削除する。

#### 受け入れ条件

- [ ] 削除前に確認メッセージを表示する
- [ ] ユーザーがキャンセルした場合は削除しない
- [ ] ユーザーが確定した場合のみ削除する
- [ ] 削除後、一覧画面から該当アイテムが消える

ユーザーはカテゴリ・ステータス・優先度でアイテムを絞り込める。

登録数が増えても、必要なアイテムを素早く見つけられるようにするため。

#### EARS形式

When ユーザーがフィルター条件を選択したとき、the system shall 条件に一致するアイテムのみを一覧表示する。

#### 受け入れ条件

- [ ] カテゴリで絞り込める
- [ ] ステータスで絞り込める
- [ ] 優先度で絞り込める
- [ ] 複数条件を組み合わせて絞り込める
- [ ] フィルターを解除できる
- [ ] 条件に一致するアイテムがない場合は、その旨を表示する

## 7.2 優先度P1：初期リリースに含めたい機能

### 10. 進捗率の自動計算機能

#### 説明

ユーザーが総量と現在の進捗を入力した場合、システムは進捗率を自動計算して表示する。

#### 目的

読書・動画・教材などの進み具合を視覚的に把握できるようにするため。

#### 計算例

読書：

- 現在ページ ÷ 総ページ数 × 100

動画：

- 視聴済み時間 ÷ 動画時間 × 100

教材：

- 現在の章 ÷ 全体の章数 × 100

#### EARS形式

When ユーザーが総量と現在の進捗を入力したとき、the system shall 進捗率を自動計算して表示する。

#### 受け入れ条件

- [ ] total_amount と current_amount が入力されている場合、進捗率を表示する
- [ ] 進捗率は0〜100%の範囲で表示する
- [ ] current_amount が total_amount を超える場合はエラーを表示する
- [ ] total_amount が0または未入力の場合は進捗率を表示しない
- [ ] 読書・動画・教材のいずれでも同じ計算ロジックを使う

### 11. 今日やるリスト機能

#### 説明

ユーザーは登録済みアイテムの中から、今日取り組むものを選んで「今日やるリスト」に追加できる。

#### 目的

登録しただけで終わらせず、その日に実行する行動へつなげるため。

#### EARS形式

When ユーザーがアイテムの「今日やる」ボタンを押したとき、the system shall そのアイテムを今日やるリストに追加する。

When ユーザーがダッシュボードを開いたとき、the system shall 今日やるリストに追加されたアイテムを表示する。

#### 受け入れ条件

- [ ] ユーザーは任意のアイテムを今日やるリストに追加できる
- [ ] 今日やるリストはダッシュボード上に表示される
- [ ] 今日やるリストからアイテムを外せる
- [ ] 今日やるリスト内でもタイトル、カテゴリ、次にやることが表示される
- [ ] 完了ステータスのアイテムは今日やるリストから外す、または完了済みとして表示する

### 12. URLを開くボタン機能

#### 説明

URLが登録されているアイテムでは、ユーザーがワンクリックで外部ページを開ける。

#### 目的

動画・教材・記事・商品ページなどにすぐアクセスできるようにするため。

#### EARS形式

When アイテムにURLが登録されているとき、the system shall URLを開くボタンを表示する。

When ユーザーがURLを開くボタンを押したとき、the system shall 新しいタブで該当URLを開く。

#### 受け入れ条件

- [ ] URLが登録されている場合のみボタンを表示する
- [ ] ボタンを押すと新しいタブでURLが開く
- [ ] URLが未登録の場合はボタンを表示しない
- [ ] 不正なURL形式の場合は保存時に警告を表示する

---

### 13. 期限切れ表示機能

#### 説明

期限が設定されており、現在日付を過ぎている未完了アイテムを期限切れとして表示する。

#### 目的

放置されている後回し案件を見落とさないようにするため。

#### EARS形式

When アイテムの期限が現在日付より前であり、かつステータスが完了またはやめた以外のとき、the system shall そのアイテムを期限切れとして表示する。

#### 受け入れ条件

- [ ] 期限切れアイテムがダッシュボードに件数表示される
- [ ] 一覧画面で期限切れアイテムが分かりやすく表示される
- [ ] 完了済みアイテムは期限切れとして扱わない
- [ ] やめたステータスのアイテムは期限切れとして扱わない
- [ ] 期限が未設定のアイテムは期限切れとして扱わない

---

### 14. 今週中に期限が来るアイテム表示機能

#### 説明

期限が今日から7日以内に設定されている未完了アイテムを表示する。

#### 目的

近いうちに対応すべきものを早めに把握するため。

#### EARS形式

When ユーザーがダッシュボードを開いたとき、the system shall 今日から7日以内に期限が来る未完了アイテムを表示する。

#### 受け入れ条件

- [ ] 今日から7日以内に期限が来るアイテムを表示する
- [ ] 完了済みアイテムは表示しない
- [ ] やめたステータスのアイテムは表示しない
- [ ] 対象アイテムがない場合は「今週中の期限はありません」と表示する

---

### 15. 空状態表示機能

#### 説明

アイテムが存在しない場合や、フィルター条件に一致するアイテムがない場合、適切な案内メッセージを表示する。

#### 目的

ユーザーが「エラーなのか、データがないだけなのか」を誤解しないようにするため。

#### EARS形式

When 表示対象のアイテムが0件のとき、the system shall 状況に応じた空状態メッセージを表示する。

#### 受け入れ条件

- [ ] 初回利用時は「まずはアイテムを登録しましょう」と表示する
- [ ] フィルター結果が0件の場合は「条件に一致するアイテムはありません」と表示する
- [ ] 今日やるリストが空の場合は「今日やるものはまだ選ばれていません」と表示する
- [ ] 空状態でも画面レイアウトが崩れない

# 8. 非機能要件

### 1. パフォーマンス

#### 要件

アプリは個人利用を想定し、通常操作がストレスなく行える速度で動作すること。

#### 受け入れ条件

- [ ] ダッシュボード表示は通常時3秒以内に完了する
- [ ] アイテム一覧表示は100件以下で3秒以内に完了する
- [ ] アイテム登録・編集・削除の反映は3秒以内に完了する
- [ ] フィルター操作は1秒以内に画面へ反映される
- [ ] 画像や動画ファイル本体は保存しないため、過度に大きなデータ通信を発生させない

---

### 2. セキュリティ

#### 要件

ユーザーごとのデータが安全に分離され、他人のアイテムを閲覧・編集できないこと。

#### 受け入れ条件

- [ ] Supabase Authを用いて認証を行う
- [ ] 各アイテムにはuser_idを紐づける
- [ ] ログイン中ユーザーは自分のuser_idに紐づくアイテムのみ取得できる
- [ ] Row Level Securityを有効化する
- [ ] 未ログインユーザーはデータ取得・登録・編集・削除を行えない
- [ ] パスワードをアプリ側で平文保存しない
- [ ] 環境変数は.env.localで管理し、GitHubに公開しない

---

### 3. 操作性

#### 要件

ユーザーが迷わず登録・確認・更新できるシンプルなUIにすること。

#### 受け入れ条件

- [ ] アイテム登録画面は1分以内に入力完了できる項目数にする
- [ ] 主要操作は3クリック以内で到達できる
- [ ] ダッシュボードからアイテム追加画面に1クリックで移動できる
- [ ] 一覧画面から編集・削除・詳細表示にすぐアクセスできる
- [ ] カテゴリ・ステータス・優先度は自由入力ではなく選択式にする
- [ ] 必須項目と任意項目が見分けられる
- [ ] エラー時は専門用語ではなく、ユーザーに分かりやすい文言で表示する

---

### 4. レスポンシブ対応

#### 要件

PCとスマホブラウザの両方で使えること。

#### 受け入れ条件

- [ ] 横幅375px程度のスマホ画面でも主要操作ができる
- [ ] PC画面ではカードまたはリストが見やすく表示される
- [ ] スマホではフォーム入力欄が画面幅に収まる
- [ ] ボタンはスマホでも押しやすいサイズにする
- [ ] 横スクロールが発生しないようにする

---

### 5. データ整合性

#### 要件

不正な入力や矛盾したデータが保存されないようにすること。

#### 受け入れ条件

- [ ] タイトルが空の場合は保存できない
- [ ] 次にやることが空の場合は保存できない
- [ ] 総量と現在の進捗は0以上の数値のみ受け付ける
- [ ] 現在の進捗が総量を超える場合は保存できない
- [ ] URL形式が不正な場合は警告を表示する
- [ ] カテゴリ・ステータス・優先度は定義済みの値のみ保存する
- [ ] 削除操作は確認後にのみ実行する

---

### 6. エラー処理

#### 要件

#### 受け入れ条件

- [ ] 削除に失敗した場合は「削除に失敗しました」と表示する

# 11. 実装時の注意

このアプリは、最初から高機能にしすぎないことを重視する。

特に以下は避ける。

- 最初からAI機能を入れる
- 最初から通知機能を入れる
- 最初から外部API連携を入れる
- カテゴリごとに複雑なテーブルを分ける
- デザインにこだわりすぎて機能実装が止まる

まずは、以下の流れが安定して動くことを最優先とする。

1. ログインする
2. アイテムを登録する
3. 一覧で見る
4. 編集する
5. 完了にする
6. フィルターする
7. ダッシュボードで状況を見る

この基本操作が完成した後に、必要に応じてAI分類・通知・外部API連携などを追加する。
